<task type="auto">
  <name>Task 1: Retire/restore lifecycle for all equipment types</name>
  <files>app/routers/equipment.py, app/templates/equipment/index.html, app/templates/equipment/_grinder_card.html, app/templates/equipment/_brewer_card.html, app/templates/equipment/_paper_card.html, app/templates/equipment/_water_card.html, app/templates/equipment/_setup_card.html, app/static/css/main.css</files>
  <action>
**Add retire/restore routes to `app/routers/equipment.py`:**

For each equipment type, add a toggle endpoint:

`POST /equipment/grinders/{id}/retire` — Set is_retired=True. Also auto-retire all BrewSetups where grinder_id matches this grinder. Redirect to /equipment.
`POST /equipment/grinders/{id}/restore` — Set is_retired=False. Redirect to /equipment.

Same pattern for:
- `POST /equipment/brewers/{id}/retire` and `/restore` (auto-retire setups using this brewer)
- `POST /equipment/papers/{id}/retire` and `/restore` (auto-retire setups using this paper)
- `POST /equipment/water-recipes/{id}/retire` and `/restore` (auto-retire setups using this water recipe)
- `POST /equipment/setups/{id}/retire` and `/restore` (direct setup retire/restore)

**Auto-retire logic:** When retiring a component, query all active BrewSetups that reference it and set their is_retired=True too. When restoring a component, do NOT auto-restore setups (user can restore them manually — some setups may have other retired components).

**Update each equipment card template** (_grinder_card, _brewer_card, _paper_card, _water_card, _setup_card):
- Add a "Retire" button (small, secondary/danger style) for active items
- Add a "Restore" button for retired items
- Retired items: add a `.retired` CSS class that makes the card visually distinct (lower opacity, gray border, "Retired" badge)

**Update equipment/index.html:**
- "Show retired" toggle at page top: when checked, page reloads with `?show_retired=true` query param
- When show_retired=true, include retired items in each section (they render with .retired styling)
- Count badge logic: active count by default; if show_retired, show "2 active, 1 retired" format

**Add CSS to main.css:**
- `.card.retired { opacity: 0.6; border-color: var(--text-muted); }` (or similar)
- `.badge-retired { background: var(--text-muted); color: var(--bg-primary); font-size: var(--text-sm); padding: 2px 8px; border-radius: var(--radius-sm); }`

**Update the wizard (GET /equipment/setups/new):** Ensure it only queries active (non-retired) equipment for selection lists. This should already be the case from Plan 04 but verify and enforce `filter(Model.is_retired == False)` on all wizard queries.
  </action>

Retire a grinder → it disappears from equipment page (default view).
Check "Show retired" → retired grinder appears grayed out with "Retired" badge.
Retire a grinder that's part of a setup → setup also auto-retired.
Restore the grinder → grinder active again; setup still retired (manual restore needed).
Wizard only shows active equipment.
  </verify>
  <done>Retire/restore works for all equipment types and setups. Auto-retire cascades to setups. Show/hide retired toggle works. Visual distinction for retired items. Wizard excludes retired equipment.</done>

  <name>Task 2: Brew page setup + bean selection UX</name>
  <files>app/routers/brew.py, app/templates/brew/index.html</files>
  <action>
**Update `app/routers/brew.py`:**

Modify the `GET /brew` route to also:
- Query active (non-retired) brew setups with joined relationships
- Read `active_setup_id` from cookie (same pattern as `active_bean_id`)
- If `active_setup_id` cookie exists, look up the setup; if not found or retired, clear it
- Pass `active_setup`, `setups` (list of all active setups) to the template
- Add a helper function `_get_active_setup(request, db)` similar to `_get_active_bean`

Add route: `POST /brew/set-setup` — Set active setup from form POST (like `set-active-bean`):
- Accept `setup_id: str = Form(...)`, validate it exists and is active
- Set `active_setup_id` cookie (1-year expiry, httpOnly, samesite=lax)
- Redirect to /brew

**Update `app/templates/brew/index.html`:**

Replace the current simple brew page with a two-panel selection layout:

**Panel 1 — Equipment Setup card:**
- If no setup selected: show a prompt card "Select your equipment setup" with a button/link that opens a dropdown/selection list of active setups
- If setup selected: show a compact setup card (name + components list, similar to _setup_card from equipment page)
- "Change Setup" button that reveals a dropdown/list of other setups
- Small "Edit Setup" link → goes to /equipment/setups/{id}/edit
- Small "New Setup" link → goes to /equipment/setups/new

**Panel 2 — Bean selection card:**
- Keep the existing bean selection behavior (active_bean cookie, bean picker)
- Show selected bean name + metadata, or "Select a bean" prompt

**Layout:**
- Side-by-side on desktop (min-width: 768px) — use CSS flexbox or grid, each panel ~50%
- Stacked on mobile — setup on top, beans below
- Both panels are `.card` elements

**Below the panels:**
- The existing brew flow (recommend button, measurements, etc.) remains, but only shows when BOTH setup and bean are selected
- If either is missing, show a clear message: "Select a setup and a bean to start brewing"

**Default selection behavior:**
- First time: prompt to select (or create if none exist)
- After first use: last-used setup and beans are pre-selected (via cookies)

The brew page should feel like: "pick your setup, pick your beans, let's go" — two clear inputs before recommendations.

Do NOT implement inline setup editing (swapping components mid-brew) in this plan — that's a future enhancement. Just allow selecting/changing the whole setup.

Touch targets: 48px+ for all selection buttons.
