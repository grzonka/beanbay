  <name>Task 2: Redesign brewer form with progressive disclosure and update card with tier badge</name>
  <files>app/templates/equipment/_brewer_form.html, app/templates/equipment/_brewer_card.html</files>
  <action>
    **Brewer form (`_brewer_form.html`):**

    Redesign the form with three sections:
    1. **Basic info** (always visible): Name + Brew Methods (existing fields)
    2. **Machine Capabilities** (expandable `<details>` element, open by default in edit mode if any non-default capability is set): Temperature, Pre-infusion, Pressure, Flow, Stop Mode, Bloom

    Use native HTML `<details>/<summary>` for progressive disclosure — no JavaScript needed, works on all browsers. The capabilities section should be labeled "Machine Capabilities" with a summary like "Configure what your machine can control".

    Structure each capability group as a fieldset-like section:

    **Temperature Control:**
    - `temp_control_type`: `<select>` with options from TEMP_CONTROL_TYPES (or hardcode: none/preset/pid/profiling). Default selected: current value or "pid" for create.
    - `temp_min`, `temp_max`, `temp_step`: number inputs, only shown/relevant when temp_control_type is not "none". Use placeholder text like "85", "96", "1.0". These don't need JS hide/show — just always show them, the labels make it clear they're for temp range.

    **Pre-infusion:**
    - `preinfusion_type`: `<select>` with options from PREINFUSION_TYPES. Default: "none".
    - `preinfusion_max_time`: number input with placeholder "15" (seconds). Label: "Max pre-infusion time (seconds)".

    **Pressure Control:**
    - `pressure_control_type`: `<select>` from PRESSURE_CONTROL_TYPES. Default: "fixed".
    - `pressure_min`, `pressure_max`: number inputs. Placeholders "6.0", "12.0".

    **Flow Control:**
    - `flow_control_type`: `<select>` from FLOW_CONTROL_TYPES. Default: "none".

    **Other:**
    - `has_bloom`: checkbox (label: "Supports bloom/soak pause")
    - `stop_mode`: `<select>` from STOP_MODES. Default: "manual".

    Use the project's existing CSS classes: `form-group`, `form-label`, `form-input`, `checkbox-item`. For selects, use class `form-input` (same as text inputs — the project doesn't have a separate select class).

    For the `<details>` element, style it with a bottom margin. The `<summary>` should use a slightly larger font and cursor: pointer.

    In edit mode, auto-open the `<details>` if the brewer has any non-default capabilities (i.e., if `brewer.temp_control_type != "pid"` or `brewer.preinfusion_type != "none"` or `brewer.pressure_control_type != "fixed"` or `brewer.flow_control_type != "none"` or `brewer.has_bloom` or `brewer.stop_mode != "manual"` or any float fields are set). Use Jinja: `{% if brewer and (brewer.preinfusion_type != "none" or ...) %}open{% endif %}`.

    For create mode (no `brewer` object), pass the enum constants from the route context OR hardcode them in the template (hardcoding is simpler and these values change rarely). **Recommendation: hardcode the select options in the template** — this avoids having to pass constants through every route that renders the form (including the equipment_index page for the add modal). The constants in the model are for Python-side validation, not template rendering.

    **Brewer card (`_brewer_card.html`):**

    Add a tier badge next to the brewer name. Use the `derive_tier` function passed in template context:

    ```html
    <div class="card-title">
      {{ brewer.name }}
      <span class="badge" style="margin-left: var(--spacing-xs); font-size: 0.75rem;">
        T{{ derive_tier(brewer) }}
      </span>
    </div>
    ```

    The badge uses the existing `badge` CSS class. The tier number (T1-T5) is compact and informative. Keep it small — it's supplementary info, not primary.

    If `derive_tier` is not available in the template context (defensive), fall back gracefully:
    ```html
    {% if derive_tier is defined %}
      <span class="badge" ...>T{{ derive_tier(brewer) }}</span>
    {% endif %}
    ```
