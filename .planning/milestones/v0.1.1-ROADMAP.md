# Milestone v0.1.1: UX Polish & Manual Brew

**Status:** ✅ SHIPPED 2026-02-22
**Phases:** 10-12
**Total Plans:** 8

## Overview

Make the app feel right on every screen, eliminate lazy-default UX traps, and let users record brews without waiting for BayBE recommendations. Mobile gets a proper hamburger/drawer nav, desktop gets a full-width sidebar layout, taste scoring requires deliberate interaction, and manual brew entry feeds directly into BayBE optimization.

## Phases

### Phase 10: Responsive Navigation & Layout ✅

**Goal:** App layout adapts to any screen — hamburger/drawer on mobile, sidebar on desktop, active bean indicator never overflows
**Depends on:** Nothing (CSS/HTML changes, independent of backend)
**Requirements:** NAV-01, NAV-02, NAV-03
**Plans:** 2 plans

Plans:
- [x] 10-01-PLAN.md — Mobile hamburger + drawer navigation
- [x] 10-02-PLAN.md — Desktop sidebar layout + responsive container

**Success Criteria:**
1. On mobile (<768px), navigation is a hamburger icon that opens a drawer/overlay with all nav links — no horizontal tab row visible
2. On desktop (≥768px), navigation is a fixed sidebar and main content uses full available width — not a 480px centered column
3. Active bean indicator displays cleanly in both mobile drawer and desktop sidebar without text wrapping or overflow, even with long bean names
4. All existing pages remain functional and visually correct after layout changes

### Phase 11: Brew UX Improvements ✅

**Goal:** Brew flow interactions are deliberate and guided — no lazy defaults, no silent dead ends
**Depends on:** Nothing (can run parallel to Phase 10)
**Requirements:** UX-01, UX-02, FLOW-01
**Plans:** 2 plans

Plans:
- [x] 11-01-PLAN.md — Inactive taste slider + submit gate (UX-01, UX-02)
- [x] 11-02-PLAN.md — No-bean prompt on /brew (FLOW-01)

**Success Criteria:**
1. Taste score slider starts inactive/greyed (opacity 0.4, untouched state) and user cannot submit the brew form until they explicitly touch/interact with it
2. When "Failed Shot" toggle is activated, taste score is overridden to 1 and the slider becomes disabled — same as existing behavior, preserved through the new inactive-start pattern
3. When user navigates to /brew without an active bean, they see a clear "Pick a bean first" message with a direct link to bean selection — no silent redirect

### Phase 12: Manual Brew Input ✅

**Goal:** User can record any brew manually with all parameters and a taste score, and it feeds into BayBE optimization
**Depends on:** Phase 11 (taste slider inactive pattern applies to manual form; no-bean prompt pattern reused)
**Requirements:** FLOW-02, FLOW-03, FLOW-04
**Plans:** 4 plans

Plans:
- [x] 12-01-PLAN.md — DB migration (is_manual) + brew page restructure + record endpoint
- [x] 12-02-PLAN.md — Manual brew form with editable params + toggleFailed extraction
- [x] 12-03-PLAN.md — Manual badge in history + batch delete with campaign rebuild
- [x] 12-04-PLAN.md — Adaptive parameter range extension

**Success Criteria:**
1. From the brew page, user can choose "Manual Input" to enter all 6 recipe parameters (grind, temp, preinfusion, dose, yield, saturation) plus taste score and submit
2. Submitted manual brew is saved and fed to BayBE via add_measurement — it counts as an optimization data point
3. Manual brews are visually distinguishable from BayBE-recommended brews in shot history (blue "Manual" badge)
4. Manual brew form validates inputs within existing parameter ranges before submission

## Milestone Summary

**Execution Order:** 10 → 11 → 12

**Stats:**
- 3 phases, 8 plans, 33 commits
- 17 files modified, +1,418 / -71 lines
- 130 tests (25+ added)
- ~8,295 total LOC
- Completed: 2026-02-22

**Key Decisions:**
- Render /brew with no_active_bean flag instead of redirect — silent redirects confusing on mobile
- data-touched attribute on taste slider (not JS var) — state co-located with DOM element
- Display "—" as initial taste slider label — 7.0 default looked like a deliberate choice
- window.toggleFailed exposed from tags.js IIFE — three templates needed same function
- POST /beans/set-active for bean picker — existing activate route uses path param
- Bounds validation only for is_manual — optimizer recs always in-bounds
- hidden(no) + checkbox(yes) for saturation — HTML checkbox submits nothing when unchecked
- fetch extend-ranges then form.submit() — ensures bounds update persists before validation

**Issues Resolved:**
- Mobile navigation scaling (too many tabs, bean name wrapping)
- Desktop layout wasting screen space (480px centered column)
- Lazy taste score defaults (pre-filled 7.0 encouraging meaningless submissions)
- Silent redirect on /brew without active bean
- No way to record manual brews outside BayBE recommendation flow

**Technical Debt Incurred:**
- Orphaned CSS rules (.nav-links, .nav-link, .nav-active-bean) from old horizontal nav — dead code (~30 lines)
- Active bean indicator in mobile drawer uses inline styles instead of CSS classes
- .badge-manual references undefined --text-xs CSS variable (falls back correctly to .badge base class)
